슬롯은 이미 만들어져 있는 프롬프트의 구조, 틀을 말한다.

[유형별 분류](https://www.notion.so/35f329e97c4880d4a4c7dc40e8519efe?pvs=21)

**계속 반복 가능한 것들**

- 기능 소개 (이미 있는 기능을 다른 각도로 계속 소개)
- 사용 사례 ("이런 사람한테 유용해요")
- 문제 제기 ("이런 문제 겪어본 적 있나요?" → 우리 서비스로 연결)
- 피드백 요청

**한 번밖에 못 쓰는 것들**

- 런치 공지
- 수치 공개 (업데이트가 없으면 변화가 없음)
- 회고/배운 것 (새로운 개발이 없으면 소재가 없음)

내가 지금 해야 하는 리서치 방향은 다음과 같다.

 스레드에서 인디해커와 1인 창업자들이 실제로 어디에 홍보글을 올리는지, 어떤 형식으로 올리는지, 어떤 유형의 게시물이 반복되는지 찾아봐야 한다.

### 리서치에서 건진 핵심 인사이트

### 1. 실제로 인디해커들이 올리는 게시물 유형

실제 데이터 기반으로 반응이 좋은 포스트 유형이 꽤 명확해요.

- **수치 투명 공개** — "$0 → $X MRR", 유저 수 업데이트. 수익/결정 과정/실패까지 공유하는 창업자가 기능만 올리는 사람보다 팔로워를 4-6배 더 모은다는 데이터가 있어요. [Prems AI](https://prems.ai/blog/indie-hacker-marketing-playbook-2026)
- **회고/배운 것** — 실패해도 "무엇을 배웠는가"가 있으면 반응이 온다
- **기능 스크린샷/GIF** — 새 기능 GIF + 짧은 캡션 같은 "빵부스러기" 형식의 뒷이야기 포스트 [Wisp CMS](https://www.wisp.blog/blog/from-0-to-1000-followers-the-strategic-path-for-indie-hackers-on-twitter)
- **스레드** — 스레드는 단일 트윗보다 54% 더 높은 인게이지먼트 [Highperformr](https://www.highperformr.ai/blog/twitter-for-indie-hackers)
- **피드백 요청/폴** — 다음 기능 투표 같은 참여 유도형
- **문제 제기형** — 제품 얘기 없이 "이런 문제 겪어봤어요?" 식의 공감 유도

### 슬롯 설계를 위해 먼저 정해야 할 것

슬롯은 결국 **"어떤 유형의 게시물이 반복 가능한가"** 목록이에요.

리서치 결과랑 인디해커 실제 패턴을 합쳐보면 이렇게 나와요.

| 슬롯 유형 | 예시 | 재사용 주기 |
| --- | --- | --- |
| 기능 소개 | "이런 기능 만들었어요" + GIF/스크린샷 | 새 기능마다 |
| 문제 제기 | "이런 문제 겪어봤나요?" | 2-3주마다 |
| 수치 공개 | "이번 달 유저 N명, MRR $X" | 월 1회 |
| 회고/배운 것 | "만들면서 이걸 깨달았어요" | 격주 |
| 피드백 요청 | "다음 기능 뭐가 필요해요?" | 월 1-2회 |
| 사용 사례 | "이런 사람이 이렇게 씁니다" | 2-3주마다 |
| 업데이트 공유 | "버그 고쳤어요 / 속도 개선했어요" | 업데이트마다 |

# 스레드 api 관련 특이사항

Threads API 인증이 Instagram을 통해 돌아가요 — 사용자가 Instagram Business 또는 Creator 계정이 Threads 프로필에 연결돼 있어야 해요. MVP에서 자동 발행까지 구현할 거면 팀 백엔드 쪽에 이걸 미리 공유해두는 게 좋을 것 같아요. 생각보다 인증 플로우가 복잡할 수 있거든요. [PostEverywhere](https://posteverywhere.ai/blog/post-to-threads-api)

그리고 API로 포스트 발행, 콘텐츠 조회, 댓글 관리, 조회수/좋아요/리포스트 등 지표 조회가 다 된다는 건 나중에 인사이트 기능 만들 때도 좋은 신호예요.
