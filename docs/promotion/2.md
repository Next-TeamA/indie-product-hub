# **원문 + 메타데이터**

가장 현실적인 방식은 이거야.

```markdown
## Ref 01

### 원문
I built a tiny tool to help solo founders keep track of their abandoned side projects...

### 링크
...

### 유형
재홍보/리마인드형

### 목적
예전에 만든 프로젝트를 다시 노출시키기

### 구조
1. 만든 배경
2. 해결하는 문제
3. 누구에게 유용한지
4. 피드백 요청

### 따라할 포인트
- 과장하지 않고 자연스럽게 소개함
- “나도 이 문제를 겪었다”로 시작함
- 마지막에 피드백 요청이 부담스럽지 않음

### 우리 서비스 적용
방치된 프로젝트를 다시 소개하는 자동 홍보글 템플릿으로 사용 가능
```

이 정도면 충분함. → 지피티에 원문 넣고 이 템플릿으로 하나하나 변환해서 맞는말인지 확인하고 맥락수정만 하면 될듯

→ 이거 30개 사례 모은다음에 지피티 한번에 돌려서 패턴 추출하면 될듯

# **4. 네가 지금 해야 할 서치 방식**

1시간 안에 완벽하게 구조화하려고 하면 또 느려짐.

그래서 이렇게 해.

## **1단계: 레퍼런스 10~15개 수집**

일단 원문 그대로 넣어.

## **2단계: 각 레퍼런스에 태그만 붙이기**

처음부터 길게 분석하지 말고 이 정도만.

```markdown
유형:
목적:
좋은 포인트:
우리 서비스에 쓸 수 있는가:
```

## **3단계: 마지막에 LLM에게 패턴 추출시키기**

예를 들어 Claude나 ChatGPT한테:

```
아래 레퍼런스들을 보고 인디해커 홍보 게시물 유형을 분류해줘.
각 유형별로 구조, 훅 패턴, CTA 패턴, 필요한 입력값을 뽑아줘.
그리고 우리 홍보 에이전트가 참고할 수 있는 promotion_reference.md 형식으로 재구성해줘.
```

이렇게 시키면 됨.

---

유형은 일단 이정도.

### **1. 기능 출시형**

새 기능이나 업데이트를 알리는 글.

### **2. 문제 해결형**

“내가 이런 문제를 겪어서 이걸 만들었다” 유형.

### **3. 재홍보/리마인드형**

이미 만든 프로젝트를 다시 소개하는 글.

너희 서비스에 특히 중요함.

### **4. 피드백 요청형**

서비스를 보여주고 의견을 구하는 글.

### **5. 성과 공유형**

사용자 수, 방문자 수, 매출, 가입자 수 등 수치를 공유하는 글.

### **6. 개발 로그/진행상황형**

오늘 무엇을 만들었는지 공유하는 글.

이 중 MVP에는 최소 3개만 넣어도 됨.

추천 우선순위는:

1. 재홍보/리마인드형
2. 문제 해결형
3. 기능 출시형
4. 피드백 요청형
5. 개발 로그형

왜냐면 너희 핵심이 **방치된 프로젝트를 다시 홍보하는 것**이니까.

# **팀원들에게 이렇게 말하면 됨**

레퍼런스를 그냥 통째로 md 파일에 넣는 방식도 가능할 것 같아요.

다만 원문만 모아두면 나중에 LLM이 매번 다시 해석해야 해서 결과가 흔들릴 수 있을 것 같습니다.

그래서 저는 원문을 넣되, 각 레퍼런스마다 최소한의 메타데이터만 붙이는 방식이 좋을 것 같아요.

예를 들면:

- 원문
- 링크
- 게시물 유형
- 목적
- 좋은 포인트
- 우리 서비스에 적용할 수 있는 부분

이 정도만 붙여두면, 나중에 LLM이 이걸 바탕으로 홍보 템플릿이나 프롬프트를 훨씬 안정적으로 만들 수 있을 것 같습니다.

# **결론**

**통째로 넣기 = 가능**

**통째로만 넣기 = 나중에 비효율**

**원문 + 짧은 태그 = 제일 현실적**

지금 1시간 리서치에서는 완벽한 분석 말고,

**레퍼런스 수집 + 최소 태깅**까지만 해도 충분해.
